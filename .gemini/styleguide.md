# Gemini Code Review スタイルガイド

## 基本設定
- **必ず日本語で回答してください**
- **コードレビューは建設的で親しみやすいトーンで行ってください**
- **具体的なコード例と修正案を提示してください**

## レビューの重点項目

### コードの品質
- **コードの可読性**: 変数名、関数名、クラス名が適切か
- **コードの重複**: DRY原則に従っているか
- **関数の責任**: 単一責任の原則が守られているか
- **コメントの品質**: 必要な箇所に適切なコメントがあるか

### セキュリティとパフォーマンス
- **入力値検証**: ユーザー入力の適切なバリデーション
- **SQLインジェクション対策**: パラメータ化クエリの使用
- **認証・認可**: 適切なアクセス制御の実装
- **パフォーマンス**: N+1問題やメモリリークの可能性

### エラーハンドリング
- **例外処理**: 適切なtry-catch文の使用
- **エラーメッセージ**: ユーザーフレンドリーなメッセージ
- **ログ出力**: 適切なログレベルでの出力

## フィードバック形式

### 問題指摘の際
```
❌ 問題: [具体的な問題点]
✅ 改善案: [修正方法とコード例]
💡 理由: [なぜその改善が必要か]
```

### 良い点の評価
```
👍 良い点: [具体的に評価する内容]
```

## 技術スタック別の指示

### JavaScript/TypeScript
- **型安全性**: TypeScriptの型を最大限活用
- **非同期処理**: Promise、async/awaitの適切な使用
- **ESLint/Prettier**: コードフォーマットの統一

### Python
- **PEP 8準拠**: Pythonコーディング規約の遵守
- **型ヒント**: 適切なtype hintsの使用
- **virtualenv**: 依存関係の管理

### Java
- **オブジェクト指向**: SOLID原則の適用
- **リソース管理**: try-with-resourcesの使用
- **並行処理**: スレッドセーフティの確保

## よく使われる追加指示

### コードレビューの深さ
- **表面的な指摘だけでなく、設計レベルの問題も指摘してください**
- **将来的な拡張性や保守性も考慮してください**
- **チーム開発での協働性を重視してください**

### 学習支援
- **なぜその書き方が問題なのか、具体的な理由を説明してください**
- **より良いアプローチがあれば、複数の選択肢を提示してください**
- **参考になる資料やドキュメントがあれば紹介してください**

### プロジェクト固有の要求
- **このプロジェクトのアーキテクチャパターンに合致しているか確認してください**
- **既存のコードベースとの一貫性を保っているか確認してください**
- **パフォーマンス要件を満たしているか評価してください**

## 実際のレビュー例

### 良いレビュー例
```
👍 良い点: エラーハンドリングが適切に実装されており、ユーザーにとって分かりやすいエラーメッセージが提供されています。

❌ 問題: この関数は複数の責任を持っているように見えます。
✅ 改善案: 以下のように分割することを検討してください：
[コード例]
💡 理由: 単一責任の原則に従うことで、テストが書きやすく、保守性が向上します。
```

## 追加の実用的な指示
- **コードの変更が既存のテストに影響を与えないか確認してください**
- **新しい機能には適切なテストが含まれているか確認してください**
- **APIの変更には適切なドキュメント更新が含まれているか確認してください**
- **データベースの変更にはマイグレーションスクリプトが含まれているか確認してください**
